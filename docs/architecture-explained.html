一个本地优先的 Agent Harness 外面包了一套 情报工作台。

这个仓库现在不是单纯的 CLI agent，也不是普通的 CRUD Web 应用。它的核心是 `packages/core` 里的 mini-agent runtime： 模型、工具、技能、会话、run trace 都在这里。`packages/server` 把这套 runtime 接入情报业务，并把零外发、溯源、审计这些红线做成工具边界。 `packages/web` 则提供操作者和管理员使用的 React 工作台。

01 / architecture

三层架构：core 提供 agent 能力，server 绑定业务边界，web 承载工作流。

理解这个项目最重要的是分清三层的责任。core 不懂“专题、密级、素材、通报”；server 不自己发明 agent 循环，而是把业务工具注入 core；web 不直接操作文件，只通过受鉴权的 API 和 SSE 展示结果。

core 是通用 harness

core 的稳定出口在 packages/core/src/index.ts。server 通过 `import { RuntimeAgent } from "mini-agent"` 使用它，而不是深入 import 内部文件。

server 是可信边界

server 负责把业务约束变成工具能力。agent 不能直接随便读 `cases/`，它只能调 server 暴露的 `search_chunks`、`read_chunk`、`cite`、`finalize_answer` 等工具。

web 是操作界面

前端没有业务权威。它只保存会话 token，所有数据来自 `/api`，流式问答来自 `/api/cases/:id/inquiries/stream`。

02 / main flow

端到端主链路：从文件进来，到带出处的结论出去。

这条链路是整个系统的骨架。素材不是直接喂给模型，而是先落盘、加工成 chunk、进入检索，再由 agent 用工具逐步检索、阅读、接地引用，最后通过 finalize 契约生成可入库的问答记录。

1

登录

`AuthService.login` 校验本地用户，发内存 token，审计 `auth.login`。

2

建专题

`CaseService.create` 写 `manifest.json`，按角色和密级约束创建。

3

上传素材

`MaterialService.ingest` 或 `ingestStream` 把原始文件写入 `materials/`。

4

加工切块

文档直接切 chunk；音视频图像走媒体管线，产出 `media.json` 和 `.chunks.jsonl`。

5

检索

BM25、dense embedding、rerank 组合，预算不足时只取相关上下文。

6

Agent 工具

agent 调 `search_chunks`、`read_chunk`、`cite`、按需媒体工具。

7

Finalize

`finalize_answer` 是最终结论唯一入口，raw final text 不直接入库。

8

审计展示

工具调用、问答创建、出站授权都入审计链，前端展示 citation 与回放。

web/src/api.ts

03 / core runtime

Core Runtime：一个迷你 Claude Code 风格的本地 agent 循环。

`packages/core` 的职责是通用 agent 能力：模型适配、工具调用、技能激活、会话持久化、run 轨迹。它不理解情报业务，所以后续业务能力都通过 extraTools 和 modelAdapter override 接进来。

RuntimeAgent.create

agent 的构造入口。它解析配置、准备模型适配器、工具注册表、技能注册表、session store、run store。现在支持 `RuntimeAgent.create(options)`，旧的 `new RuntimeAgent(...)` 路径已废弃。

runtime/agent.ts runtime/config.ts skills/registry.ts

RuntimeConversation.send

同一个 conversation 的 send 被内部队列串行化，避免多次 ask 同时改 `messages` 和 session JSONL。每次 send 产生一个 run，由 RunManager 管理状态和 trace。

sendQueue session JSONL runId

runAgentLoop

循环步骤是固定的：构造 system prompt，发模型请求，记录模型响应，如果有 tool_calls 就逐个执行工具，把结果追加为 tool message，再进入下一轮。没有 tool_calls 时结束。

04 / server harness

Server Harness：把业务系统拆成可审计、可组合、可降级的服务。

`createApp` 是 server 的装配中心。它决定数据根、创建审计服务、用户存储、模型适配器、OfflineGuard、模型槽、业务服务，再统一挂到 `/api` 路由下。

服务装配顺序为什么重要

`createApp` 先创建数据路径和审计，再创建鉴权和专题，再读模型配置和 slot 配置，最后把 `OfflineGuard`、`MaterialService`、`InquiryService` 等组合起来。这样每个业务服务都能复用同一条审计链和同一套访问控制。

API 路由是薄层

路由文件基本只做 HTTP 参数解析和响应。实际业务都落在 service 中，这让测试可以绕开 HTTP，直接用临时 dataDir 和 mock adapter 跑端到端。

05 / agent inquiry

Agent 问答新范式：让 agent 自由检索，但只给它安全工具。

旧问答是单次 `generateJson`：先选上下文，再让模型一次性吐 JSON，然后事后过滤 citation。三期转向后，问答进入 core harness：模型可以多轮决定检索什么、读哪些 chunk、何时引用、何时结束，但它看不到任意文件系统写工具，也不能直接把自由文本当结论入库。

tool-contractper ask ledger
createCitationLedger()
  retrieved: Map<chunk_id, Chunk>
  cited: Map<chunk_id, Citation>
  finalize: { claims } | null
  readBytes: number

extraTools for this run:
  search_chunks(query, k)
    → 检索本专题 chunks
    → 写 ledger.retrieved

  read_chunk(chunk_id)
    → 只能读 retrieved 里已有的 chunk
    → 受 readBudgetBytes / perReadCapBytes 限制

  cite(chunk_id, claim)
    → 复用 resolveValidCitations
    → 校验 content_hash
    → 写 ledger.cited

  finalize_answer({ claims })
    → 写 ledger.finalize
    → server 从这里生成 Inquiry

为什么要有 `finalize_answer`

core loop 的天然终点是“模型发了一条没有 tool_calls 的 assistant message”。这对普通聊天没问题，但对情报系统有风险：模型可能在最终自由文本里加入没有 cite 的新断言。

所以 server 侧规定：最终问答记录只从 `finalize_answer({claims})` 和 ledger 生成。没有 finalize，就判 `insufficient` 或 `error`；raw final text 不入库。

工具模型看到的能力server 实际做的约束为什么安全
search_chunks 按 query 搜索本专题片段。 固定 caseId，不接受跨专题参数；只返回 id、snippet、locator、material_name。 agent 不能枚举全局文件，也不能凭空 cite 未检索 chunk。
read_chunk 读取某个 chunk 全文。 只能读本次 ledger.retrieved 中已有 chunk；读取总量和单次长度受预算限制。 防止长文档撑爆本地模型上下文，也防止绕过检索范围。
cite 把 claim 绑定到 chunk。 通过 `resolveValidCitations` 校验 chunk 存在且 `sha256(text) === content_hash`。 citation 不是模型说了算，而是 server 根据真实 chunk 重建。
transcribe/caption_frame/ocr_region 检索不足时按需重看原始媒体。 只能读本专题素材，体量封顶，real slot 出站前过 OfflineGuard。 按需多模态产出的临时 chunk 也进入 ledger，后续仍需 cite。
finalize_answer 提交本次最终 claims。 server 只认其中 claim 和 cite_ids，未 cite 的结论降级待核或 insufficient。 把最终答案从“自由文本”变成“可检查结构”。
当前 P3.A/P3.B 问答没有把 core 默认的 `bash`、`write`、`edit` 暴露给情报 agent。P3.0-iso 的降权子进程隔离只有在未来真的给 agent 任意写工具时才需要做。
06 / invariants

三条红线不是提示词口号，而是代码里的不变量。

项目最核心的安全思路是：不要指望模型“自觉”。模型可以自由推理，但它拿到的工具、工具的参数、工具返回的数据、最终入库的格式，都由 server 侧硬约束。

Redline 1

零外发

文本模型、embedding、rerank、ASR、VLM、OCR 这些应用自身出站都要过 `OfflineGuard.authorize`。白名单 host 放行并审计，非白名单抛 403 并审计 `egress.deny`。

Redline 2

Citation 溯源

每个 chunk 带 `content_hash = sha256(text)`。模型只能提出 cite 请求，server 用真实 chunk 重建 Citation。hash 不一致或未检索过，就不进入台账。

Redline 3

审计链

审计事件稳定序列化后计算 `payload_hash`，再和前一个事件的 `prev_hash` 拼成 `event_hash`。`verify()` 可以定位篡改或断链。

流式问答 UI 的状态机

`InquiryPanel` 保存 `history`、`liveQuestion`、`liveText`、`toolTrace`、`busy`、`frameCite`。token 事件追加到 liveText；tool_start 增加 running 轨迹；tool_result 更新 ok/failed；done 后把 inquiry 加到 history 并清 live state。

引用回放不是装饰

音频 citation 带 timecode 时，点击 chip 会拉原始素材并跳播片段。视频/图像 citation 带 bbox 或 timecode 时，点击会取帧并叠加框选。这是“溯源可复核”的前端体现。

为什么不用数据库

当前目标是本地、离线、可检查。文件结构让审计、素材、模型中间结果和报告产物都可以直接被人工复核。测试也更简单：每个用例创建临时 dataDir，跑完删除。

并发靠小粒度队列

最容易出错的是 `manifest.json` 的 read-modify-write，例如同时 ingest 和 process。`CaseService` 用 core 提供的 `FileMutationQueue` 按文件路径串行，避免 last-writer-wins。

09 / roadmap
