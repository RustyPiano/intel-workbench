三条红线 · 服务层强制

▣ 零外发 OfflineGuard

应用层出站闸门，仅白名单 host（模型端点）放行，allow/deny 全入审计；生产置空即全断。配合部署层 air-gap。

▣ 审计哈希链

append-only，payload_hash · prev_hash · event_hash 单写串行，可独立 verify 定位断链。

▣ Citation 溯源

每条 claim 必须引用检索到的 chunk 且 sha256==content_hash，否则降级待核或拒答。

⚠

当前缺口：媒体（音频 / 视频 / PDF / Office）目前一律降级为 pending——只登记、不真正抽取内容。这正是 §02 推荐架构要补的部分。文本专题闭环已完整可用。

02 推荐架构 · RAG 检索 + 多模态入库 Target

两条扩展，都复用现有 Citation / 审计 / OfflineGuard 红线，一行不改：(A) 把"必检索"改成按 token 预算路由——小数据走全上下文，大数据/知识库才检索；(B) 媒体从"降级 pending"换成入库时真处理成带时间码的可引用 chunk。

A · RAG 检索路由（溯源与检索解耦）

路由判据  ·  专题切块总量 ≤ 模型上下文预算(取窗口 ~50–60%) ?
↳ 是 · 全上下文 grounding（默认）小数据最优

所有 chunk（带 id）直接喂模型 → 模型作答引用 chunk_id → 事后验哈希。零检索召回风险：每个片段都在场，不会因漏检而漏引。直接化解"RAG 压制回答质量"的顾虑。

↳ 否 · 检索路（超限 / 跨专题KB）规模兜底

BM25 ⊕ dense 经 RRF(k=60) 融合 → top-N → 可选重排（按候选数门控）→ top-k 喂模型，同样引用 chunk_id + 验哈希。

嵌入 Qwen3-Embedding-0.6B重排 Qwen3-Reranker-0.6BRRF 零依赖

向量库：单专题用进程内暴力余弦（精确、零依赖、贴合文件优先）；跨专题 KB 用 sqlite-vec（纯 C 单文件、SIMD 精确 KNN）。对溯源系统而言"精确不漏"比 ANN 速度更重要——避免索引悄悄丢掉那一个匹配块。

03 音频方案 · 阿里 FunASR（已选定） Decided

工业级工具包，一行接好端到端：VAD + ASR + 时间戳 + 说话人分离 + 标点。中文极强、部署简单、含 openai-compatible API 与 vLLM 加速。ASR 引擎可在 SenseVoice-Small（~170× 实时、最快）/ Fun-ASR-Nano（LLM 高精度多语）/ Qwen3-ASR-1.7B（52 语自动检测）间切换——正好落在我们的"ASR 槽"。

# FunASR 一行接好（端到端，含 diarization）
from funasr import AutoModel
 
model = AutoModel(
  model="iic/SenseVoiceSmall" # 或 Fun-ASR-Nano-2512 / Qwen3-ASR-1.7B —— ASR 槽
  vad_model="fsmn-vad",   # 语音活动检测
  spk_model="cam++",     # 说话人分离（关键）
  device="cuda")
 
result = model.generate(input="meeting.wav")
# → [00:00.4 → 00:03.8] Speaker 0: 文本内容 ...　→ 落为带时间码+说话人的 chunk
✓

为什么贴合本项目：① 原生 词/段级时间戳 → 直接喂 locator.timecode 满足时间码溯源；② 说话人分离（cam++）情报价值高；③ 全程在 OfflineGuard 后、本地权重、无出站；④ ASR 引擎可热插拔，先 SenseVoice 跑通、需要更高精度再换 Qwen3-ASR-1.7B / Fun-ASR-Nano。务必拿真实截获音频（噪声/口音）做头对头再定引擎。
